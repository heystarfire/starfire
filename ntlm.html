Kerberos & NTLM authentication

Kerberos


Kerberos provides a centralized authentication server whose function is to authenticate users to servers and servers to users. In Kerberos Authentication server and database is used for client authentication. Kerberos runs as a third-party trusted server known as the Key Distribution Center (KDC). Each user and service on the network is a principal.

The main components of Kerberos are

  • Authentication Server (AS): The Authentication Server performs the initial authentication and ticket for Ticket Granting Service. 
  • Database: The Authentication Server verifies the access rights of users in the database. 
  • Ticket Granting Server (TGS): The Ticket Granting Server issues the ticket for the Server 

Kerberos Authentication

  1. User Login:
    • The user logs in to their computer and requests access to a network service.
  1. Request for Ticket-Granting Ticket (TGT):
    • The user's computer asks the Authentication Server (AS) for a Ticket-Granting Ticket (TGT). This ticket is needed to request access to other services.
  1. Authentication Server Verifies and Responds:
    • The Authentication Server (AS) checks the user's credentials against its database.
    • If the credentials are correct, the AS sends back a TGT and a session key, both encrypted with the user's password.
  1. User Decrypts and Requests Service Ticket:
    • The user's computer decrypts the TGT using the password.
    • The user’s computer then sends the TGT to the Ticket Granting Server (TGS) to request a service ticket for the specific service they want to access.
  1. Ticket Granting Server Issues Service Ticket:
    • The Ticket Granting Server (TGS) decrypts the TGT, verifies the request, and issues a service ticket for the requested service.
  1. User Accesses the Service:
    • The user sends the service ticket, along with an authenticator (which includes information like the user’s name and network address), to the target service server.
    • The service server verifies the ticket and authenticator.
    • If everything checks out, the service server grants access to the user.

NTLM and its types

NTLM (New Technology LAN Manager) is a suite of authentication protocols developed by Microsoft to confirm the identity of users and protect the integrity and confidentiality of their activities on a network.

Authentication takes place through direct communication between a client and the target server, commonly known as NTLM challenge/response

Simplified explanation of how NTLM authentication works:

  1. Logon Attempt: You enter your username and password on your computer.
  1. Hash Creation: The client creates a hash of the user's password using the NTLM hashing algorithm (which involves MD4). The plaintext password is discarded after hashing.
  1. Initial Request: The client sends a request to the server with the user's username in plaintext.
  1. Challenge: The server responds with a challenge, which is a 16-byte random number.
  1. Response Generation: The client uses the hash of the password to encrypt the challenge. This encrypted challenge is sent back to the server.
  1. Server Validation: The server forwards the challenge, the client's response, and the username to the Domain Controller (DC).
  1. DC Validation: The DC retrieves the user’s stored hash from its database. It then uses this hash to encrypt the challenge it received from the server.
  1. Verification: The DC compares the encrypted challenge it generated with the one provided by the client. If they match, the authentication is successful. If not, access is denied.

This process ensures that passwords are never transmitted in plaintext, but it's important to note that NTLM is considered less secure compared to more modern authentication protocols, such as Kerberos

NTLM (NT LAN Manager)

NTLM is a challenge-response authentication protocol developed by Microsoft. It's used to authenticate users in a network environment. NTLM is designed to be used with Windows systems but can also work with other systems.

NTLMv1 (NT LAN Manager version 1)

NTLMv1 is the original version of the NTLM protocol. It was used in Windows NT 4.0 and earlier versions.

  • Process:
    1. Client Request: The client sends a logon request to the server.
    1. Challenge: The server responds with a random challenge (a nonce).
    1. Response: The client encrypts the challenge with a hash of the user's password and sends the encrypted result back to the server.
    1. Verification: The server also encrypts the challenge with the hash of the user's password stored in its database and compares it with the response received from the client. If they match, the user is authenticated.

NTLMv2 (NT LAN Manager version 2)

NTLMv2 is an improved version of NTLMv1, introduced in Windows 2000. It enhances security with stronger cryptographic measures.

  • Process:
    1. Client Request: The client sends a logon request, including a timestamp and additional data.
    1. Challenge: The server responds with a challenge and a nonce.
    1. Response: The client uses the nonce and a stronger hashing algorithm to generate a response. This response includes more information such as a session key.
    1. Verification: The server performs the same calculations using its stored password hash and compares it to the response from the client.

Hash Types and Protocols

  • LM Hash: Older, less secure hash.
  • NTLM Hash: More secure than LM.
  • NTLMv1: Uses a challenge-response mechanism but is vulnerable to various attacks.
  • NTLMv2: An improved version with enhanced security features.
